Every message you publish to DRMQ lands in a **topic** — a named, append-only log stored durably on disk. Think of a topic as an infinite tape: producers always write to the end, and consumers read forward through it at their own pace using a numeric marker called an **offset**. Because the log is persistent, messages survive broker restarts, and any consumer can revisit past messages simply by rewinding its offset.

## What Is a Topic?

A topic is the fundamental unit of organization in DRMQ. It is an ordered sequence of messages identified by a human-readable name (for example, `orders`, `payments`, or `sensor-readings`). There are no partitions — the topic itself is the single, strictly ordered log.

<CardGroup cols={2}>
  <Card title="Append-only" icon="arrow-down-to-line">
    Producers always append to the tail of the log. Existing messages are never modified or deleted before the retention window expires.
  </Card>
  <Card title="Durable" icon="hard-drive">
    DRMQ writes every message to a custom Write-Ahead Log (WAL) before acknowledging the producer. Messages survive broker crashes and restarts.
  </Card>
  <Card title="Strictly ordered" icon="list-ol">
    Every message within a topic has a unique, monotonically increasing offset. Order is guaranteed — message `n` is always older than message `n+1`.
  </Card>
  <Card title="Multi-reader" icon="users">
    Any number of consumer groups can read the same topic independently. Each group maintains its own offset pointer without affecting others.
  </Card>
</CardGroup>

## What Is an Offset?

An offset is a **sequential, monotonically increasing integer** that identifies a message's exact position in a topic. The first message appended to a topic receives offset `0`, the second receives offset `1`, and so on. Offsets never reset and never reuse a value.

When you call `producer.send(topic, payload)`, the broker returns a `SendResult` that carries the offset assigned to your message:

```java
try (DRMQProducer producer = new DRMQProducer("localhost:9092")) {
    producer.connect();
    DRMQProducer.SendResult result = producer.send("orders", "{ \"id\": 42 }");

    if (result.isSuccess()) {
        System.out.println("Message stored at offset " + result.getOffset());
    } else {
        System.err.println("Send failed: " + result.getErrorMessage());
    }
}
```

On the consumer side, each `ConsumedMessage` exposes the same offset value through `msg.offset()`, letting you correlate a processed record back to its exact position in the log.

## How Consumers Track Position

A consumer's current position is the offset of the **next message it expects to receive**. After processing offset `5`, the consumer's position advances to `6`. DRMQ stores this position on the broker so that a restarting consumer can resume exactly where it left off, without any local state file.

You control whether the broker advances that position automatically after each poll, or only when you explicitly tell it to.

## Auto-Commit vs. Manual Commit

<Tabs>
  <Tab title="Auto-Commit">
    Call `setAutoCommit(true)` before connecting. After every successful `poll()`, DRMQ automatically commits the offset of the last message in the batch. This is the simplest mode and works well when losing a few messages on consumer crash is acceptable.

    ```java
    DRMQConsumer consumer = new DRMQConsumer("localhost:9092", "order-processors");
    consumer.setAutoCommit(true);
    consumer.connect();
    consumer.subscribe("orders");

    while (true) {
        List<DRMQConsumer.ConsumedMessage> messages = consumer.poll(100, 1000);
        for (DRMQConsumer.ConsumedMessage msg : messages) {
            process(msg); // offset is committed automatically after poll
        }
    }
    ```

    <Warning>
      If your consumer crashes between `poll()` returning and your business logic completing, auto-commit may have already advanced the offset, meaning those messages will not be redelivered. Use manual commit if exactly-once processing matters to your application.
    </Warning>
  </Tab>
  <Tab title="Manual Commit">
    Leave `autoCommit` at its default (`false`) and call `consumer.commit(topic, offset)` only after you have successfully processed a message. The broker will not advance the offset until you do, so any unprocessed message is redelivered after a consumer failure.

    ```java
    DRMQConsumer consumer = new DRMQConsumer("localhost:9092", "order-processors");
    // autoCommit is false by default
    consumer.connect();
    consumer.subscribe("orders");

    while (true) {
        List<DRMQConsumer.ConsumedMessage> messages = consumer.poll(100, 1000);
        for (DRMQConsumer.ConsumedMessage msg : messages) {
            process(msg);
            // Commit offset + 1 so the next poll starts at the following message
            consumer.commit("orders", msg.offset() + 1);
        }
    }
    ```

    <Note>
      Pass `offset + 1` to `commit()` — the argument is the offset of the **next** message to consume, not the one you just processed.
    </Note>
  </Tab>
</Tabs>

## Replaying Messages from a Specific Offset

DRMQ stores messages durably on disk for the configured retention window (default: 7 days). You can replay any portion of a topic by subscribing with an explicit starting offset using `subscribe(topic, fromOffset)`.

<Steps>
  <Step title="Disable group mode">
    Switch to single consumer mode so the broker does not override your offset with the group's committed position.

    ```java
    DRMQConsumer consumer = new DRMQConsumer("localhost:9092", "replay-job");
    consumer.setGroupMode(false);
    consumer.connect();
    ```
  </Step>
  <Step title="Subscribe at the target offset">
    Pass the exact offset you want to start from. Use `0` to replay from the very beginning of the topic.

    ```java
    // Replay everything from the start of the topic
    consumer.subscribe("orders", 0);

    // Or start from a specific offset you recorded earlier
    consumer.subscribe("orders", 1_500);
    ```
  </Step>
  <Step title="Poll and track progress manually">
    Read messages and commit your position explicitly so you can resume if interrupted.

    ```java
    while (true) {
        List<DRMQConsumer.ConsumedMessage> messages = consumer.poll(100, 1000);
        for (DRMQConsumer.ConsumedMessage msg : messages) {
            System.out.printf("Replaying offset %d: %s%n",
                msg.offset(), msg.payloadAsString());
        }
        if (!messages.isEmpty()) {
            long lastOffset = messages.get(messages.size() - 1).offset();
            consumer.commit("orders", lastOffset + 1);
        }
    }
    ```
  </Step>
</Steps>

<Tip>
  The two `subscribe` overloads are `subscribe(String topic)` — which resumes from the broker-committed offset in group mode or offset `0` in single mode — and `subscribe(String topic, long fromOffset)`, which always overrides to the offset you provide. Use the second form any time you need deterministic replay.
</Tip>

## Persistent Storage

DRMQ writes every incoming message to a Write-Ahead Log before returning a success response to the producer. Log data is organized into fixed-size segments (default 100 MB each) that are retained for a configurable window (default 7 days). When a segment ages out, DRMQ removes it safely without disrupting active consumers. This means:

- Broker restarts do not lose any acknowledged messages.
- You can replay historical data up to the retention boundary.
- In cluster mode, the Raft snapshot mechanism compacts the log while preserving correctness.
