DRMQ is a high-performance, fault-tolerant distributed message broker built from first principles. It uses a full implementation of the Raft consensus algorithm for leader election and log replication, giving you strong ordering guarantees and high availability without relying on external coordination services. Whether you're running a single node for local development or a three-node production cluster with automatic failover, DRMQ exposes the same clean producer and consumer API throughout.

## Deployment Modes

DRMQ ships with two deployment modes so you can match the setup to your environment:

- **Single-node mode** — start one broker process, zero configuration beyond a port and data directory. Ideal for local development, CI pipelines, and simple deployments that don't require fault tolerance.
- **Cluster mode** — start three or more broker nodes, each aware of its peers. Raft elects a leader, replicates every write to a quorum, and automatically steps down an isolated leader to prevent split-brain data loss. Clients connect to any node and are transparently redirected to the current leader.

## Architecture Overview

DRMQ is structured as a multi-module Maven build. Each module has a clearly defined responsibility, making the codebase straightforward to navigate and extend:

- **`drmq-protocol`** — Protocol Buffers (`.proto`) definitions that describe every message exchanged between clients and brokers, and between broker peers during Raft replication. This module is compiled first so the generated Java classes are available to all other modules.
- **`drmq-broker`** — The core server. Contains the Raft node logic, a Netty-based TCP server, the segment-based Write-Ahead Log (WAL) storage engine, and atomic consumer offset management.
- **`drmq-client`** — The Java client library. Exposes `DRMQProducer` and `DRMQConsumer` with automatic reconnects, randomized bootstrap server selection, and transparent leader-redirection on `NOT_LEADER` responses.
- **`drmq-integration-tests`** — End-to-end tests that spin up real broker instances to verify multi-node scenarios, DLQ routing, and failover behaviour.
